# Architecture Decision Records (ADR) — servys

Здесь фиксируем **почему** приняли то или иное архитектурное решение (спека отвечает на «что и как»,
ADR — на «почему X, а не Y»).

## Конвенция

- **1 ADR = 1 решение.** Атомарно. Не мешаем несколько решений в один файл.
- **Имя файла:** `NNNN-краткое-название.md`, `NNNN` — сквозной порядковый номер (0001, 0002, …).
- **Не редактируем принятые ADR по существу** — если решение изменилось, заводим новый ADR
  со статусом `принято`, а старый переводим в `заменено ADR-XXXX`.
- **Каждый ADR ссылается на секцию спеки**, которую обосновывает — так ADR и спека не разъезжаются.
- **Поле `Area`** для навигации: `backend | frontend | integration | cross-cutting`.

### ⚠️ Про `cross-cutting`

Сквозные решения — это наши **швы**: HTTP API-контракт, порты `Recommender` и `Sink`, модель
тенанта b2c/b2b. Они принадлежат всем троим сразу. **Не прячь их в область одной платформы** —
помечай `Area: cross-cutting`, иначе самые важные решения окажутся «ничьими».

## Как завести новый ADR

1. `cp docs/adr/template.md docs/adr/NNNN-название.md`
2. Заполни, укажи `Area`, `Статус`, ссылку на §спеки.
3. Добавь строку в индекс ниже (в свою область).

### Кандидаты в ADR (черновой план, не решения)

Ориентир, что стоит зафиксировать — заполняется по мере принятия:

- **cross-cutting:** заморозка API-контракта · порты `Recommender`/`Sink` · модель тенанта b2c/b2b · contract-first разбивка на 3 потока.
- **backend:** HTTP-роутер (stdlib `net/http` vs chi/gin) · выбор БД (Postgres vs SQLite vs in-memory) · конфиг/секреты (env).
- **integration:** стратегия данных (гибрид LLM+кэш) · модель Claude и способ вызова · Bitrix-канал MVP (входящий вебхук + `im.message.add` vs бот `imbot.message.add`).
- **frontend:** Vue + TypeScript standalone · сборка/деплой фронта.
