# servys — дизайн MVP и командный процесс

- **Дата:** 2026-07-11
- **Команда:** 3 разработчика
- **Срок:** ~полдня (~5 рабочих часов)
- **Статус:** утверждён, источник правды для реализации

---

## 1. Продукт

**servys** — сервис для превентивного обслуживания авто.
Зная марку/модель/год и пробег, он подсказывает, что и когда нужно обслужить:

- **регламентные работы** по периодичности (например, замена масла раз в 7000–10000 км);
- **типовые поломки**, характерные для конкретной модели, привязанные к пробегу.

Основное представление сервиса — **самостоятельное веб-приложение (Vue)**. Bitrix24 —
**не UI, а интеграционный канал**: уведомления, а в перспективе — CRM-движок удержания клиентов.
Ядро (Go-бэкенд) автономно, поэтому к нему можно подключать разные представления и каналы.

### Два режима (tenant type)

- **b2c** — частный автовладелец. Работает через Vue-приложение; опционально получает
  напоминания в Bitrix24.
- **b2b** — автосервис/дилер/автопарк. Тот же основной процесс через Vue **плюс** интеграция
  с их Bitrix24 CRM: подошло ТО клиента → автоматически лид/сделка/задача в воронке.
  Это главный «продаваемый» угол: знание о ТО превращается в генератор лидов внутри CRM,
  где клиент уже работает.

b2b — это **аддитивное** расширение b2c (тот же продукт + включённый CRM-синк), а не отдельная система.

## 2. Архитектура

Ядро на Go + отдельный веб-фронт. Разделяем два типа представлений сервиса:

- **Тянущие (pull) — потребители HTTP API.** Vue-фронт дёргает API и рисует. Это основное
  представление **для обоих режимов**. Потом сюда же — мобилка/бот.
- **Толкающие (push) — синки, куда пишет сам бэкенд.** IM-уведомления, события календаря,
  сделки/задачи в CRM. Это исходящие адаптеры за портом `Sink`.

```
                 ┌───────────────────────────────┐
   Vue-фронт ──▶ │   HTTP API (pull, общий)       │
 (b2c и b2b)     │   ЯДРО (Go, tenant-aware)      │
                 │   Recommend(car) -> Result     │
                 └──────────────┬────────────────┘
                                │ порт Sink (push, по типу тенанта)
             ┌──────────────────┼─────────────────────┐
             ▼                  ▼                     ▼
        IM-уведомления     Календарь            CRM: сделки/задачи
        (оба, MVP)         (оба, v2)            (ТОЛЬКО b2b) ⭐
```

- **Go-бэкенд** — ядро: HTTP JSON API, доменные модели, хранилище, движок рекомендаций.
  LLM-интеграция живёт **Go-пакетом внутри этого же модуля** (`recommender/`), а не отдельным сервисом.
- **Vue-фронт (TypeScript)** — **отдельный сервер, standalone веб-приложение** (не iframe), ходит в Go API.

Данные («модель → регламент / поломка на пробеге») формируются по **гибридной** схеме:
LLM (Claude API) генерирует рекомендации → кладём в БД как кэш/базу знаний → можно править вручную.
Внешний готовый датасет — **исследуем позже**, схема БД должна это допускать.

### Швы под будущий рост (закладываем в скелет сразу)

Чтобы путь `уведомления → календарь → CRM/b2b` был дописыванием, а не переписыванием:

1. **Ядро рекомендаций stateless и tenant-agnostic:** `Recommend(car) -> Result` — один движок на оба режима.
2. **Исходящий порт `Sink`** рядом с `Recommender`: b2c → IM, b2b → CRM. Новый канал = новый адаптер.
3. **`Tenant{Type: b2c|b2b}`** в доменной модели с самого начала (для MVP — один b2c-тенант).
4. **Пробег — временной ряд** (замеры км с датой), а не одно число: нужно для due-detection,
   оценки даты в календаре и подпитки от автопарков.

## 4. Замороженные контракты

Это **швы** между разработчиками. Их согласуем и фиксируем в первые ~40 минут (фаза 0),
после чего каждый работает против контракта, а не против чужого живого кода.

### A. HTTP API-контракт (Go-бэк ↔ Vue-фронт, pull — общий для обоих режимов)

```
GET  /api/v1/health
  resp: {"status":"ok"}

POST /api/v1/recommendations
  req:  {"make":"Toyota","model":"Camry","year":2018,"mileage_km":95000}
  resp: {
    "car": {"make":"Toyota","model":"Camry","year":2018,"mileage_km":95000},
    "items": [
      {
        "id":"oil",
        "title":"Замена моторного масла",
        "category":"regular",          // "regular" | "known_issue"
        "severity":"medium",           // "low" | "medium" | "high"
        "interval_km":10000,            // 0, если неприменимо
        "due_at_km":98000,
        "status":"due_soon",           // "overdue" | "due_soon" | "upcoming"
        "note":"..."
      }
    ],
    "generated_by":"llm",              // "llm" | "cache"
    "cached": false
  }
```

`frontend/mock/recommendations.json` содержит валидный пример этого ответа —
Dev 2 рисует UI по нему, пока живой API не поднят.

### B. Доменные типы + порт `Recommender` (Dev 1 ↔ Dev 3)

Общие типы — в пакете `domain` (заморожены на фазе 0):

```go
package domain

type TenantType string

const (
    TenantB2C TenantType = "b2c"
    TenantB2B TenantType = "b2b"
)

type Tenant struct {
    ID   string
    Type TenantType
    // BitrixCreds, EnabledSinks — добавим на этапе b2b
}

type Car struct {
    Make      string
    Model     string
    Year      int
    MileageKm int
}

type Item struct {
    ID         string
    Title      string
    Category   string // "regular" | "known_issue"
    Severity   string // "low" | "medium" | "high"
    IntervalKm int    // 0 если неприменимо
    DueAtKm    int
    Status     string // "overdue" | "due_soon" | "upcoming"
    Note       string
}

type Result struct {
    Items       []Item
    GeneratedBy string // "llm" | "cache"
    Cached      bool
}
```

```go
package recommender

type Recommender interface {
    Recommend(ctx context.Context, car domain.Car) (domain.Result, error)
}
```

- **Dev 1** пишет API против **заглушки** `Recommender` (возвращает хардкод-`Result`).
- **Dev 3** реализует боевой вариант: Claude API + кэш в `store`, подменяет заглушку.
- Модель Claude выбираем на этапе реализации (сверяемся со скиллом `claude-api`);
  дефолт для скорости/качества демо — Sonnet 5, ключ через env `ANTHROPIC_API_KEY`.

### C. Исходящий порт `Sink` (Dev 1 определяет ↔ Dev 3 реализует в `bitrix/`)

Единый интерфейс для всех push-каналов. В MVP реализуем только IM; календарь и CRM — те же
`Deliver`, но адаптер решает по типу тенанта.

```go
package sink

type Reminder struct {
    Tenant domain.Tenant
    Car    domain.Car
    Items  []domain.Item // просроченные/скоро наступающие ТО
}

type Sink interface {
    Deliver(ctx context.Context, r Reminder) error
}
```

- **Dev 1** определяет порт `Sink` и вызывает его из ядра; в MVP wiring подключает Bitrix-IM-адаптер.
- **Dev 3** реализует `Sink` в `bitrix/`: MVP — `im.message.add` (или бот `imbot.message.add`);
  v2 — `calendar.event.add`; v3 (b2b) — `crm.item.add`/`crm.activity.todo.add`/`tasks.task.add`.

## 5. Архитектура фронта (Dev 2)

Фронт — тонкий слой над контрактом A: получить `car` из формы, дёрнуть API, отрисовать `items`.
Вся сложность — в чётком разделении слоёв, чтобы работать против мока и переключиться на живой
API сменой env, а не правкой кода.

**Стек:** Vue 3 (`<script setup>` + Composition API) + **TypeScript (strict)** + Vite.
Состояние — локальные composables (Pinia/Vuex для одного экрана избыточны). Сеть — нативный
`fetch` в тонкой обёртке. Пакетный менеджер — npm.

### Раскладка `frontend/`

```
frontend/
├── index.html
├── vite.config.ts          # dev-proxy /api → Go-бэк (локально обходим CORS)
├── .env / .env.production  # VITE_API_BASE_URL, VITE_USE_MOCK
├── mock/recommendations.json
└── src/
    ├── main.ts
    ├── App.vue             # layout + оркестрация экрана
    ├── types/api.ts        # TS-типы контракта A (1:1 с mock.json) — единственный источник форм данных
    ├── api/client.ts       # fetch-обёртка: postRecommendations(car), health(); mock/live-переключатель
    ├── composables/
    │   └── useRecommendations.ts   # статус запроса idle|loading|success|error + data
    ├── components/
    │   ├── CarForm.vue             # ввод make/model/year/mileage + валидация
    │   ├── RecommendationList.vue  # список + состояния loading/empty/error
    │   └── RecommendationCard.vue  # карточка item: severity/status → бейдж/цвет
    └── ui/
        ├── tokens.css              # CSS-переменные: палитра, отступы, радиусы, типографика
        └── status.ts               # маппинг severity/status → лейбл + цвет (единое место)
```

### Слои и ответственность

1. **`types/api.ts` — контракт как типы.** Ровно повторяет ответ `POST /api/v1/recommendations`
   (union-литералы для `category`/`severity`/`status`). Всё типизируется от него; расхождение
   с `mock.json` правим здесь — это единственный источник правды по форме данных на фронте.
2. **`api/client.ts` — единственная точка сети.** Компоненты не знают про `fetch`/URL. Базовый
   URL из `VITE_API_BASE_URL`; флаг `VITE_USE_MOCK=1` отдаёт `mock/recommendations.json` без сети
   (Dev 2 работает, пока API не поднят). Переключение на живой API = смена env, а не кода.
3. **`useRecommendations.ts` — состояние экрана.** Держит статус (`idle/loading/success/error`)
   и результат; вся асинхронщина и обработка ошибок инкапсулированы здесь.
4. **Компоненты — презентационные.** `CarForm` эмитит `submit(car)`, `App.vue` зовёт composable,
   `RecommendationList`/`Card` рисуют. Бизнес-логики в компонентах нет.

### UI/UX

- **Один экран:** форма ввода → под ней список рекомендаций (или пустое/лоадинг/ошибка).
- **Дизайн-токены** в `tokens.css` (без тяжёлого UI-фреймворка): единая палитра, типографика,
  отступы. Светлая тема, адаптив от мобильного.
- **Семантика статусов — единый маппинг в `status.ts`:**
  - `severity`: `low` → нейтральный, `medium` → янтарный, `high` → красный;
  - `status`: `overdue` → красный «Просрочено», `due_soon` → янтарный «Скоро», `upcoming` → серый «Впереди».
  - Список сортируем: сперва `overdue`, затем `due_soon`, затем `upcoming`.
- **Состояния списка:** скелетон/спиннер при загрузке, дружелюбное пустое состояние, явная
  ошибка с кнопкой «Повторить».
- **Форма:** `year`/`mileage_km` — числа с валидацией; кнопка `disabled`, пока форма невалидна.

### Конфиг и локальная разработка

- **Dev-proxy в `vite.config.ts`:** `/api` → `http://localhost:<go-port>`, чтобы в деве не ловить
  CORS; в проде фронт и API за одним доменом/реверс-прокси.
- **CORS на бэке:** для standalone-фронта Go-API отдаёт `Access-Control-Allow-Origin`
  (согласовать с Dev 1 на фазе 0). Секретов на фронте нет — ключ Claude только на бэке.

### Швы под рост (в MVP не реализуем, но не мешаем)

- API-слой изолирован → мобилку/бота позже прикрутят к тем же типам из `types/api.ts`.
- `tenant` фронту в b2c-MVP не нужен; при появлении b2b добавится в запрос/контекст, не ломая экран.
- Пробег сейчас одно число в форме; когда бэк перейдёт на временной ряд — добавится отдельный
  виджет истории, форма ввода не меняется.

## 9. Скоуп MVP и роадмап

**MVP (сегодня, режим b2c):**
- поток: ввёл марку/модель/год/пробег → список рекомендаций (Vue standalone);
- рекомендации от LLM с кэшем в БД;
- **демо-уведомление в Bitrix24** (IM-сообщение по просроченным ТО) — синк работает end-to-end;
- **одна демо-машина отрабатывает железобетонно**.

**Роадмап (аддитивно, поверх тех же швов):**
- **v2:** календарь Bitrix (`calendar.event.add`) + оценка даты по среднему пробегу;
- **v3 (b2b):** OAuth-приложение, multi-tenant, CRM-движок удержания (лиды/сделки/задачи).

**Отложено (не делаем сегодня):**
- аккаунты и сохранение нескольких авто;
- планировщик рассылки (в MVP — ручной триггер синка);
- внешний готовый датасет (только исследование источников);
- UI-редактор базы знаний;
- каталог/автокомплит моделей (сегодня — свободный ввод текста).

## 10. Как команда работает с этим документом и с Claude Code

- Документ и `CLAUDE.md` **закоммичены в репо** → у каждого появляются после `git pull`.
- Каждый участник запускает Claude Code **внутри своего клона `servys/`**.
  `CLAUDE.md` в корне подхватывается автоматически при старте сессии — то есть «натравливать»
  Claude на спеку вручную не нужно, контекст уже загружен. Достаточно сказать, например:
  *«Я Dev 2, выполни мой чек-лист фронтенда»*.
- На этапе плана этот дизайн разбивается на **3 коротких чек-листа** (по одному на разработчика).
  Каждый работает от своего чек-листа, а не от общей формальной спеки.
- Опционально: у кого установлен плагин Superpowers — использует скиллы
  `executing-plans` / `test-driven-development` поверх своего чек-листа.
